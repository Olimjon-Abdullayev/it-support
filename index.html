01

Printers

Most printer tickets are spool service, driver, or network path issues — in that order of likelihood.

🖨 Not printing — general triage order
Run top to bottom, stop at first fix
01
Check the printer's own panel for errors first — paper jam, low toner, offline light, tray empty. Physical state beats software every time.
02
Confirm it's not paused or set offline in Windows: Settings → Printers & scanners → select printer → make sure "Use printer offline" is unchecked.
03
Restart the Print Spooler service (see fix below) — fixes the majority of "stuck queue" tickets.
04
Ping the printer's IP from the user's PC. No reply = network/switch port issue, not printer software.
05
Check if it prints a test page from the printer's own panel. If that fails, it's a hardware/printer-side issue, not the PC.
06
Try printing from a different machine on the same network. Isolates whether it's the one PC or the printer/network.
07
Remove and re-add the printer using the correct driver (not "Generic / Text Only").
⚙ Restart the Print Spooler (fixes stuck queues)
1
Press Win + R, type services.msc, Enter.
2
Find Print Spooler → right-click → Restart.
3
If restart is greyed out / it won't stop: open an elevated CMD and run net stop spooler then net start spooler.
4
If the queue is still stuck, clear it manually before restarting: delete all files in C:\Windows\System32\spool\PRINTERS (stop the spooler first, then delete, then start it again).
One-liner (admin CMD/PowerShell) net stop spooler & del /Q /F "%systemroot%\System32\spool\PRINTERS\*.*" & net start spooler
📋 Print queue stuck on one job
1
Cancel the job from the queue first — if it's stuck "Deleting," that confirms a spooler-level lock, not a printer issue.
2
Restart the Print Spooler service (above). This almost always clears a "Deleting" job that won't go away.
3
If it reappears every time, the print spool folder has a corrupted .SHD/.SPL file pair — stop the spooler, manually delete files in the PRINTERS spool folder, restart spooler.
4
Large jobs (big PDFs, high-res images) can simply be slow to spool — wait 60s before assuming it's stuck.
🔌 Printer shows "Offline" in Windows
1
Uncheck "Use Printer Offline" (right-click printer in Devices & Printers).
2
Confirm the printer's IP hasn't changed — network printers are commonly added by static IP; if the printer got a new DHCP lease, Windows is now pointing at a dead address. Check printer panel → Network info, compare to the port configured in Printer Properties → Ports.
3
Ping the printer IP. Timeout = it's a network/power/cable problem at the printer, not Windows.
4
For USB-connected printers: try a different USB port/cable, and check it isn't in USB selective suspend (Device Manager → printer's USB hub → Power Management).
Best practiceFor office network printers, always assign a DHCP reservation or static IP on the printer itself — saves recurring "offline" tickets after router reboots.

📄 Paper jams
1
Always pull jammed paper in the direction of normal paper travel — pulling backward can tear paper and leave scraps inside.
2
Check all trays, the duplexer unit (back/bottom panel), and the fuser area — most jams hide in one of these three.
3
Repeated jams from one tray = worn pickup/feed rollers (common on machines printing 50k+ pages) or paper loaded above the fill line / wrong paper weight.
4
Fan and re-square the paper stack before loading — humidity-stuck sheets are a very common cause.
🌐 Can't add a network printer / printer not found
1
Confirm the PC and printer are on the same subnet/VLAN — printers on a guest or isolated VLAN won't be discoverable.
2
Add manually instead of relying on discovery: Add Printer → "The printer that I want isn't listed" → Add using TCP/IP address → enter printer IP, port type RAW (9100) or use the right driver.
3
Check Windows Firewall isn't blocking mDNS/Bonjour (UDP 5353) or printer discovery, especially on domain-joined laptops with strict firewall profiles.
4
For print-server-hosted printers, confirm the print server is reachable: \\printservername in Explorer.
📠 Scan-to-email / scan-to-folder not working (MFPs)
1
Scan-to-folder: confirm the share permissions still allow the printer's service account write access — password rotations on that account are the #1 cause.
2
Scan-to-email: check the SMTP relay settings on the printer's web admin page — port, auth, and TLS settings must match what the mail server now requires (orgs that migrate to Microsoft 365 often need app passwords or a relay connector).
3
Test by browsing to the printer's IP in a browser to reach its embedded web server — most config and clear error logs live there, not on the panel.
02

Network Issues — Laptops & PCs

Work outward: NIC → local stack → switch/AP → DHCP/DNS → upstream. Don't jump to "the network is down" before checking the one machine.

📶 "No internet" — general triage order
01
Check the network icon in the tray — globe/exclamation icon tells you a lot before you type anything.
02
Run ipconfig /all. Look at the IPv4 address: 169.254.x.x means no DHCP response (APIPA) — go straight to DHCP troubleshooting below.
03
Ping the default gateway: ping <gateway-ip>. Fails = local link/switch/Wi-Fi problem, not internet.
04
Ping a public IP (e.g. ping 8.8.8.8). Works but websites don't load = DNS issue, not connectivity.
05
Try nslookup google.com. Fails = DNS resolution broken; works = likely a proxy, certificate, or browser-specific issue.
06
Check if it's just this device: ask another user nearby, or check if their phone on Wi-Fi has the same problem.
07
Re-seat/replace the patch cable and try a different switch port before condemning the NIC.
⌨ Essential CMD/PowerShell commands
CommandWhat it does
ipconfig /allFull adapter info: IP, subnet, gateway, DNS, MAC, lease
ipconfig /releaseDrop current DHCP lease
ipconfig /renewRequest new DHCP lease
ipconfig /flushdnsClear local DNS resolver cache
ping <ip/host>Test reachability + latency
ping -t <ip>Continuous ping (Ctrl+C to stop) — good for spotting intermittent drops
tracert <host>Show the hop-by-hop path; find where latency/loss starts
nslookup <host>Test DNS resolution directly
netstat -anoList active connections + owning process ID (pair with Task Manager "Details" tab)
arp -aShow local ARP table (IP↔MAC mappings learned)
netsh winsock resetReset the Winsock catalog (fixes some "no internet despite valid IP" cases) — reboot after
netsh int ip resetReset TCP/IP stack entirely — reboot after
getmacShow MAC addresses of all adapters
route printShow local routing table
🚫 IP is 169.254.x.x (APIPA / no DHCP)
Means the PC never got a lease from a DHCP server
1
Re-seat the network cable / toggle Wi-Fi off-on. Sounds dumb, fixes a surprising number of these.
2
Run ipconfig /release then ipconfig /renew as admin.
3
Check the switch port LED — amber/off vs green/blinking tells you if link is even established at layer 1.
4
If it's one port only: move the cable to a known-good port. If that works, the original port or its run is bad.
5
Confirm the switch port isn't in a dead/disabled VLAN, or that the DHCP scope on that VLAN isn't exhausted (ask network team to check scope utilization).
6
For Wi-Fi: forget the network and rejoin, and confirm the device isn't stuck on a captive portal needing re-auth.
📡 Wi-Fi connected but "No internet" / limited connectivity
1
Forget the Wi-Fi network and reconnect with the password — clears a stale/corrupt profile, the #1 cause.
2
Check if other devices on the same AP have internet — isolates AP/uplink vs. single-device issue.
3
Update the Wi-Fi adapter driver — outdated drivers commonly mishandle DHCP/DNS on enterprise WPA2-Enterprise networks.
4
Disable then re-enable the adapter in Device Manager.
5
If 2.4GHz works but 5GHz doesn't (or vice versa), it's an AP/band/channel issue — try forcing the other band temporarily.
6
Check for a MAC filtering / NAC (network access control) policy if it's a new or replaced device that's never connected before.
🐢 Slow network / intermittent drops
1
Run ping -t to the gateway for a few minutes while reproducing — look for spikes or timeouts, not just average latency.
2
Check Device Manager for the NIC — yellow warning icon or "this device is having problems" = driver issue.
3
On a wired connection, check duplex/speed mismatch: NIC properties → Configure → Speed & Duplex — should be "Auto Negotiation," forcing it usually causes more problems than it solves unless matching a known-misbehaving switch port.
4
Bad/marginal cable is extremely common for "intermittent" issues specifically — swap the patch cable before deep diagnostics.
5
Check for a flapping port on the switch side (ask network team for port error counters: CRC errors, collisions).
6
Rule out bandwidth hogs locally: Task Manager → Performance → check if backup/sync software (OneDrive, Backup agents) is saturating the link.
🔒 VPN won't connect / drops
1
Confirm base internet works first (VPN rides on top of it — don't troubleshoot VPN on a dead connection).
2
Check system clock/date — large clock drift breaks certificate-based VPN auth (common after a dead CMOS battery, see Hardware section).
3
If on a hotel/public/guest Wi-Fi: some block the VPN's required ports (often UDP 500/4500 for IPsec, or TCP 443 for SSL VPN) — try mobile hotspot to confirm.
4
Restart the VPN client service, then try a clean reinstall if it persists — corrupted virtual adapter config is common after Windows updates.
5
Double-NAT or home router VPN passthrough settings can block site-to-site IPsec — check if disabling "VPN passthrough" weirdly fixes it (router bug, but it happens).
🔗 Cable plugged in, no link light / "unidentified network"
1
No link light at all on either end = bad cable, bad port, or disabled port. Swap cable first — cheapest test.
2
"Unidentified network" with a link light = layer 1 is fine but DHCP/layer 3 failed — go to DHCP/APIPA flow above.
3
Check the adapter isn't disabled in Device Manager or via a physical Wi-Fi/Ethernet toggle/function key on the laptop.
4
Docking stations: re-seat the dock fully, and check the dock's Ethernet works for another laptop to isolate dock vs. laptop NIC.

Cabling Tactics

Patch cord colors, wiring standards, and the easiest ways to remember pinouts on site.

🎨 Standard patch cable colors — what they mean (common convention, not a hard rule)
Color-coding is a site convention, not an IEEE standard — confirm local convention, but this is the most widely used scheme
ColorTypical use
● GreenStandard user workstation drops
● Yellow / OrangeVoIP phones / phone-only drops
● BlueServer-to-switch / patch panel-to-switch links
● PurpleVLAN-tagged or special-purpose links
● RedCritical/restricted infrastructure, security systems
● Grey / WhiteCross-connects, uplinks between switches/racks
● BlackOften outdoor-rated / building-to-building runs
Reality checkEvery site invents its own scheme. The only universal rule: pick one convention, label it on a sheet near the patch panel, and stay consistent. When in doubt, check existing cabling before introducing a new color.
🧷 RJ45 wiring standards — T568A vs T568B
The easiest method: memorize "Green and Orange swap places, everything else is identical."
T568B (most common in the US / most offices)
Pin 1White/Orange
Pin 2Orange
Pin 3White/Green
Pin 4Blue
Pin 5White/Blue
Pin 6Green
Pin 7White/Brown
Pin 8Brown
T568A (US gov / some legacy + residential)
Pin 1White/Green
Pin 2Green
Pin 3White/Orange
Pin 4Blue
Pin 5White/Blue
Pin 6Orange
Pin 7White/Brown
Pin 8Brown
Memory trickSay it as a chant: "Orange, Orange, Green, Blue, Blue, Green, Brown, Brown" for T568B pins 1-8 (stripe/solid pairs share the color). Blue and Brown pairs never move between standards — only Orange and Green swap.
Rule of thumbUse the same standard on both ends for a straight-through cable (T568B↔T568B). Only mix A on one end and B on the other if you intentionally want a crossover cable (rarely needed now — modern NICs auto-sense, so straight-through cables work almost everywhere, including PC-to-PC).
🛠 Crimping an RJ45 connector — step by step
1
Strip ~1.5cm (½–¾") of outer jacket with the crimper's stripper or a cable stripper. Don't nick the inner conductors.
2
Untwist the 4 pairs, straighten the 8 wires, and arrange in order per T568A or T568B (decide standard before you start, match both ends).
3
Trim all 8 wires flat and even, leaving about 1.3cm (½") straight before the jacket.
4
Slide wires into the RJ45 plug, copper-side up/facing the clip side, fully seated against the front — jacket should also slide in slightly past the crimp point for strain relief.
5
Verify wire order through the clear plastic before crimping — this is the step everyone skips and regrets.
6
Insert into crimper and squeeze firmly until it clicks/bottoms out. This seats the pins through the insulation and locks strain relief.
7
Test with a cable tester before deploying — wire-map test catches reversed pairs, opens, and shorts in seconds.
📏 Cable categories at a glance
CategoryMax speedMax distance (rated speed)Notes

Rule of thumbFor a typical office desktop or VoIP phone drop, Cat6 is the current sweet spot — cheap enough, fast enough, and handles PoE well. Use Cat6a for anything that might need 10GbE later (server racks, future-proofed risers).
🔍 Using a cable tester (wire-map test)
1
Plug the main unit into one end, the remote/terminator into the other.
2
Run the test — it lights up pins 1-8 in sequence on both ends.
3
Sequential 1-2-3-4-5-6-7-8 on both ends = good straight-through cable.
4
Split pairs (e.g. 1-2 lights up out of order) will often still "work" for basic connectivity but cause crosstalk and intermittent errors under load — always re-terminate, don't just shrug it off.
5
"Open" on a pin = broken/missing connection at one end — re-crimp that end.
6
"Short" between two pins = wires touching inside the connector — re-crimp, check for jacket strip damage.
🏷 Labeling & management — habits that save future-you
•
Label both ends of every cable run with the same ID (e.g. matching patch panel port + wall jack number).
•
Keep a simple spreadsheet/photo of the patch panel-to-switch port mapping — saves hours during outages.
•
Velcro ties, not zip ties, in racks — zip ties deform cable jackets over time and are a pain to redo.
•
Don't exceed the cable's bend radius (roughly 4x the cable diameter) — tight bends behind patch panels are a classic source of mystery packet loss.
•
Keep copper data cable away from running parallel too close to power cable for long stretches — induced interference is real at scale (it's a non-issue for a 1.5m desk patch cord).

IP Cameras & IP Phones

Both are network appliances first — most "broken" cameras/phones are actually PoE, VLAN, or DHCP problems.

📷 IP camera offline / no video feed
1
Check the physical LED on the camera (if present) — solid/blinking patterns usually indicate link and PoE status; consult the model's LED legend.
2
Confirm PoE is actually being delivered: check the switch port's PoE status (most managed switches show per-port PoE draw in their web UI/CLI) — a camera pulling 0W means no power, not a video software issue.
3
Ping the camera's IP. No reply + PoE confirmed = the camera itself may have crashed; power-cycle by disabling/re-enabling the switch port (avoids unplugging in tight spots).
4
Confirm it's on the correct VLAN — cameras are very often segmented onto a dedicated surveillance VLAN, and a misconfigured port (wrong/missing VLAN tag) silently kills connectivity even though link lights look fine.
5
Check the NVR/VMS software itself: is it just one camera tile gone grey, or all of them? All-down points to NVR/recording server or network core, not the individual camera.
6
If reachable by ping but the stream won't load, check the camera's RTSP/HTTP port isn't blocked by an ACL/firewall rule that changed recently.
⚡ PoE basics & troubleshooting
StandardMax power at PDTypical use
802.3af (PoE)~12.95WBasic IP phones, simple cameras, access points
802.3at (PoE+)~25.5WPTZ cameras, higher-power APs
802.3bt (PoE++/UPOE)~51-71WHigh-end PTZ, thin clients, large displays
1
Confirm the switch port is actually PoE-capable and PoE is enabled (some managed switches let you disable PoE per-port).
2
Check total PoE budget on the switch isn't exhausted — adding one more high-draw PTZ camera can tip a switch over its power budget, causing it to refuse power to new devices.
3
Cable run too long or poor quality cable can cause voltage drop sufficient to make PoE unreliable — stay within rated distance (100m) and use decent quality cable for PoE runs.
4
If using a PoE injector instead of a PoE switch port, confirm the injector itself is powered and rated for the device's wattage class.
☎ IP phone won't register / no dial tone
1
Check the phone's screen for a status/registration message — most IP phones display "Registering," "Unregistered," or a specific error code directly.
2
Confirm it has an IP address (phone's network/status menu) — same APIPA/DHCP logic as a PC applies.
3
Confirm it's on the correct voice VLAN — phones usually need a specific VLAN ID pushed via LLDP/CDP from the switch; if that's misconfigured the phone may sit on the wrong (data) VLAN and fail to reach the call manager/SIP server.
4
Check PoE delivery the same way as cameras above — a phone with a blank screen often simply isn't getting power.
5
Confirm the call control server (e.g. SIP registrar, Call Manager, 3CX, etc.) shows the extension registered, and check the time/date on the phone — large clock skew breaks SIP digest auth on some systems.
6
Factory reset as a last resort if config looks corrupted — but back up/note the extension and any speed dials first.
🔇 One-way audio / choppy audio / no audio on calls
1
One-way audio is classically a NAT/firewall RTP traversal issue — media (RTP) and signaling (SIP) take different paths and one is being blocked, especially across VPNs or when calling external numbers.
2
Choppy/garbled audio on an otherwise fine network = packet loss or jitter on the voice VLAN — check QoS is correctly prioritizing voice traffic, and check for bandwidth contention (someone running a big upload on the same link).
3
Check the headset/handset itself isn't the issue — swap headsets, test with handset only.
4
Confirm correct codec negotiation if it's a SIP trunk issue — mismatched codecs between provider and PBX cause garbled or failed audio on some calls only.
🌙 Camera image quality issues

05

Microsoft Office Issues

Outlook profile corruption and Office activation/licensing account for most repeat tickets.

📧 Outlook won't open / crashes on launch
1
Launch in safe mode first to isolate add-ins: hold Ctrl while clicking Outlook, or run outlook.exe /safe from Run.
2
If safe mode works fine, a faulty add-in is the cause — File → Options → Add-ins → manage COM Add-ins → disable all, re-enable one at a time.
3
Repair the Office installation: Control Panel → Programs → Microsoft Office → Change → Quick Repair first, Online Repair if that doesn't fix it.
4
Corrupted profile is common after a bad shutdown — create a new Outlook profile: Control Panel → Mail (Microsoft Outlook) → Show Profiles → Add, then set as default.
5
Run outlook.exe /resetnavpane to fix a corrupted navigation pane (folder list disappeared/frozen).
6
For "something went wrong" generic crash loops, check for a corrupted .OST file — close Outlook, rename the .OST in %localappdata%\Microsoft\Outlook, let Outlook rebuild it on next launch (this re-downloads mail, can take a while on large mailboxes).
📤 Mail stuck in Outbox / send-receive errors
1
Check the message itself isn't oversized (attachment over org/mailbox limit) — split into smaller files or use a cloud link instead.
2
Send/Receive → Update Folder, and check the error code shown in Send/Receive Progress for specifics.
3
Disable any antivirus/DLP add-in temporarily to rule out it intercepting and blocking the send silently.
4
Check Outlook isn't in Work Offline mode (Send/Receive tab, toggle).
🔎 Outlook search not returning results
1
Check the search scope dropdown — it may be set to "Current Folder" instead of "All Mailboxes/All Outlook Items."
2
Rebuild the Windows Search index: Control Panel → Indexing Options → Advanced → Rebuild. Can take hours on a large mailbox — warn the user.
3
Confirm Outlook is included in the indexed locations list (Indexing Options → Modify).
🔑 Office activation / licensing errors
1
Check sign-in account: File → Account in any Office app — confirm it's signed in with the correct work/school account, not a personal Microsoft account.
2
"Unlicensed Product" banner: usually a temporary token expiry — sign out and back in, or run ospp.vbs /act from the Office install directory (admin CMD) to force re-activation against the KMS/online licensing service.
3
Confirm the user actually has an assigned Microsoft 365 license in the admin portal — licenses can be unassigned during offboarding/role changes by mistake.
4
Multiple Office versions installed (e.g. leftover 2016 + new 365) commonly cause activation conflicts — fully uninstall old versions using the Microsoft Support and Recovery Assistant (SaRA) tool before reinstalling.

🔁 Office keeps prompting for credentials repeatedly
1
Open Credential Manager (Control Panel) → Windows Credentials → remove any saved entries related to Office/Outlook/OneDrive, then sign in fresh.
2
Check for an expired or changed password not yet updated in cached credentials — common right after a forced password reset.
3
Conditional Access / MFA policy changes on the tenant side can also trigger repeated prompts — check with the M365 admin if this is happening org-wide.
4
Modern Authentication must be enabled tenant-wide for some sign-in flows to behave — note for escalation if this is widespread, not single-user.
☁ OneDrive sync problems
1
Click the OneDrive cloud icon in the tray — it shows current sync status and specific errors directly, check there first.
2
Pause and resume sync (right-click tray icon) — clears many transient stuck-sync states.
3
Look for files with a red "x" or sync-pending icon — usually a filename issue (illegal characters: \ / : * ? " < > |), a file open elsewhere, or path length over 400 characters.
4
If broadly broken: unlink the account (Settings → Account → Unlink this PC) then sign back in — rebuilds sync state from scratch.
5
Check available disk space — OneDrive silently stops syncing new files when local disk is nearly full.

PC Hardware Issues

The most common desk-side hardware tickets, ranked by frequency.

⚡ PC won't power on at all
1
Check the wall outlet and power cable with a known-good device/cable — dead outlets and frayed cables are extremely common and instantly solve the "ticket."
2
Check the PSU's physical switch (back of desktop case) is in the ON (1) position, not OFF (0).
3
Listen/look for any sign of life — fans spinning briefly, lights flickering, then dying = PSU or short, vs. truly nothing = check power delivery first.
4
Laptops: remove battery if removable, hold power button 30s with everything unplugged, then try AC-only with no battery — isolates a swollen/dead battery from a charging circuit issue.
5
Desktop: disconnect all peripherals and non-essential internal components (extra RAM stick, GPU if onboard video exists) and try a minimal boot — isolates a short caused by a failing component.
🟦 Blue Screen of Death (BSOD) — common codes
Stop codeUsual cause
MEMORY_MANAGEMENTFaulty RAM, or driver corrupting memory — run Windows Memory Diagnostic / memtest86
IRQL_NOT_LESS_OR_EQUALBad driver, often network or storage driver — boot to safe mode, roll back/update the driver
DRIVER_IRQL_NOT_LESS_OR_EQUALSame as above — specific driver named in the error if shown
PAGE_FAULT_IN_NONPAGED_AREARAM, storage, or driver issue — check disk health (chkdsk) and RAM
CRITICAL_PROCESS_DIEDCore system file corruption — run sfc /scannow and DISM /Online /Cleanup-Image /RestoreHealth
UNMOUNTABLE_BOOT_VOLUMEBoot drive corruption/failing drive — check disk health urgently, may need recovery from boot media
WHEA_UNCORRECTABLE_ERROROften hardware-level — failing RAM, CPU, or overheating; check temps and run memory test
VIDEO_TDR_FAILUREGPU driver crash/timeout — update or roll back GPU driver, check for overheating
Quick diagnosticOpen Event Viewer → Windows Logs → System, filter for Critical/Error around the crash timestamp — the BugCheck event usually names the exact faulting driver/module.
🐌 PC running slow — triage order
1
Open Task Manager → Performance tab. Check CPU, Memory, Disk usage at a glance — whichever is pegged near 100% tells you where to focus.
2
Disk at 100% with low CPU/memory = classic sign of a failing or near-full HDD/SSD, or a misbehaving background process (check Task Manager → Disk column, sort descending).
3
Check Startup tab (Task Manager) for excessive startup apps — disable non-essential ones.
4
Check free disk space — Windows gets noticeably sluggish below roughly 10-15% free space on the boot drive (paging file and temp space get squeezed).
5
Check for pending Windows Updates stuck installing in the background, or a full antivirus scan running unexpectedly.
6
Check drive health: run wmic diskdrive get status or use manufacturer SSD/HDD tool — a drive throwing SMART warnings explains "everything is slow" perfectly.
🌡 Overheating / random shutdowns
1
Check vents aren't blocked (desk clutter, laptop on a soft surface/bed/lap blocking intake).
2
Open the case/check fans are spinning — dust buildup on heatsinks/fans is the single most common cause over a device's lifetime.
3
Compressed air clean of fans, heatsinks, and vents — do this outdoors/away from the PC, hold fans still while blowing to avoid generating reverse voltage.
4
Check thermal paste age on older systems (3+ years) — dried out paste causes gradual thermal throttling/shutdowns under load.
5
Monitor temps under load with HWMonitor/HWiNFO — sudden shutdown near 90-100°C CPU confirms thermal protection kicking in.
🖥 No display / "No Signal" on monitor
1
Confirm the monitor is on the correct input source (HDMI1 vs HDMI2 vs DP) — very common false alarm.
2
Swap the video cable — cheap and rules out the most common physical-layer cause.
3
If a dedicated GPU is installed, confirm the cable is plugged into the GPU, not the motherboard's onboard video port (easy mistake after a GPU upgrade).
4
Listen for POST beep codes if available — single beep is normal, patterns of beeps indicate specific hardware faults (consult motherboard manual for the exact pattern).
5
Reseat the RAM (most common single cause of POST failure) — remove, blow out the slot, firmly reseat until clips click.
🔋 Laptop battery won't charge / drains fast
1
Check the charging brick and cable on another laptop if possible — chargers fail more often than the laptop's charging circuit.
2
Check Windows Battery Report for wear level: run powercfg /batteryreport in CMD, open the generated HTML — compares Design Capacity vs Full Charge Capacity to show real degradation %.
3
Update chipset/battery drivers — Device Manager → Batteries, and check OEM power management software is current.
4
"Plugged in, not charging" stuck at a fixed percentage is often a Windows battery driver glitch — uninstall the "Microsoft ACPI-Compliant Control Method Battery" device in Device Manager and let Windows reinstall it on reboot.
5
Physically swollen battery = stop using immediately, flag for hardware replacement — fire/safety risk, don't just note it for later.
🕐 Clock keeps resetting / BIOS settings don't save
1
Classic sign of a dead CMOS battery (small coin cell on the motherboard, usually CR2032).
2
Replace the CMOS battery — cheap, common, and fixes random BIOS resets, wrong clock causing cert/VPN failures, and "boot device not found" after power loss in one shot.
3
After replacing, re-enter any custom BIOS settings (boot order, virtualization toggle, etc.) since they reset to default.
💾 Failing hard drive / SSD — warning signs
SignMeaning
Clicking/grinding noise (HDD)Mechanical failure imminent — back up immediately, this is urgent
Frequent freezes during file accessBad sectors or failing controller — check SMART status
"S.M.A.R.T. failure imminent" at bootDrive is reporting its own pending failure — back up now, schedule replacement
Files randomly corruptingFailing sectors silently corrupting writes — back up and replace
Very slow boot, fine once loadedCould simply be a slow HDD vs SSD expectation — but check SMART to rule out failure first
Priority ruleAny SMART warning or unusual drive noise = back up data before further troubleshooting, not after. Drives showing warning signs can fail completely within hours to weeks.
🔌 USB device not recognized / USB port not working
1
Try a different USB port, and a different known-good device in the same port — isolates port vs. device.
2
Check Device Manager for a yellow warning triangle or "Unknown Device" — right-click → Update Driver, or uninstall and re-plug to force redetection.
3
Disable USB selective suspend in Power Options (Advanced settings) — known to cause devices to drop out intermittently, especially on laptops.
4
For USB-C docks: confirm the dock's own firmware/drivers are current, and that the laptop's USB-C port supports the dock's full feature set (some ports are charge-only or data-only).
⌨ Keyboard / mouse not working
1
Wireless: swap/check batteries first, then check the USB receiver is firmly seated (try a different port).
2
Bluetooth: remove and re-pair the device — stale pairing is the most common cause of "randomly stopped working."
3
Wired: test on another machine if possible to isolate device vs. PC port/driver.
4
Check for a physical mute/disable switch (some keyboards have a wireless toggle switch on the underside).
5
If only some keys fail on a wired keyboard: liquid damage or debris under those specific keys — common, often not user-reported until asked directly.
🧠 Suspected bad RAM
1
Symptoms: random BSODs, especially MEMORY_MANAGEMENT; random reboots/freezes under load; failure to POST with beep codes.
2
Run Windows Memory Diagnostic (search "Windows Memory Diagnostic" in Start) or boot Memtest86 from USB for a more thorough pass — leave it running multiple passes overnight for intermittent issues.
3
With multiple sticks installed, test one stick at a time in the same slot to isolate which stick (or which slot) is bad.
4
Reseat all RAM regardless — a surprising number of "bad RAM" tickets are actually just a stick that worked itself slightly loose.

OSI & TCP/IP Models

The framework behind every "is it the network or the app" question. Memorize the layer order, and you can localize almost any fault fast.

🧩 The OSI Model — 7 layers
Mnemonic top-down: "All People Seem To Need Data Processing" · bottom-up: "Please Do Not Throw Sausage Pizza Away"
7
Application
What the user/software directly interacts with — the actual network service being used.
HTTP, HTTPS, FTP, SMTP, DNS, SSH, RDP
6
Presentation
Translates/formats data: encryption, compression, character encoding between systems.
SSL/TLS, JPEG, ASCII, encryption
5
Session
Establishes, manages, and tears down the connection/session between two devices.
NetBIOS, PPTP, session tokens, RPC
4
Transport
End-to-end delivery, segmentation, flow control, and error recovery between hosts.
TCP (reliable), UDP (fast/no guarantee), ports
3
Network
Logical addressing and routing between different networks — this is where IP addresses live.
IP, ICMP, routers, routing tables
2
Data Link
Physical addressing (MAC) and how data is framed/transmitted on the local segment.
Ethernet, MAC address, switches, ARP
1
Physical
The actual physical medium: cables, radio signals, voltages, connectors.
Cat6 cable, fiber, Wi-Fi radio, hubs, RJ45
Practical use on tickets"No internet" almost always means start checking at Layer 1 (link light?) → Layer 2 (correct VLAN/MAC?) → Layer 3 (got an IP? can you ping the gateway?) → Layer 4 (right port open?) → Layer 7 (does the actual app/site work?). Walking up the stack in order avoids skipping the obvious cause.
🧱 TCP/IP Model — 4 layers (the practical, real-world model)
OSI is the teaching model; TCP/IP is what the actual internet runs on. Mapping below.
TCP/IP LayerMaps to OSI layersExamples
Application7, 6, 5 (App/Presentation/Session)HTTP, HTTPS, DNS, FTP, SMTP, SSH
Transport4 (Transport)TCP, UDP
Internet3 (Network)IP, ICMP, ARP*
Link / Network Access2, 1 (Data Link/Physical)Ethernet, Wi-Fi, MAC addressing, cabling
Note*ARP is technically a bridge between Layer 2 and 3 — it resolves an IP address (L3) to a MAC address (L2). Don't stress the exact placement; what matters on the job is knowing it's "the thing that maps IP to MAC on the local segment."
⚖ TCP vs UDP — when each is used
TCPUDP
ConnectionConnection-oriented (handshake first)Connectionless, just sends
ReliabilityGuaranteed delivery, retransmits lost packetsNo guarantee, no retransmission
SpeedSlower (overhead of acknowledgment)Faster, lower overhead
OrderPackets arrive in orderNo ordering guarantee
Typical useWeb (HTTP/S), email, file transfer, RDPVoIP, video streaming, DNS queries, online gaming
Why this matters on the jobChoppy VoIP calls during network congestion are often "expected" behavior of UDP prioritizing speed over reliability — this is exactly why QoS is configured to protect voice traffic specifically, rather than relying on TCP-style retransmission.
🤝 TCP 3-way handshake
1
SYN — client sends a synchronize request to the server to initiate a connection.
2
SYN-ACK — server acknowledges and sends its own synchronize request back.
3
ACK — client acknowledges the server's request. Connection is now established and data can flow.
Troubleshooting relevanceIf a packet capture shows SYN sent but no SYN-ACK returned, the destination port is likely blocked by a firewall somewhere in the path — not a server application crash.

Windows, Accounts & Active Directory

Login and account lockout tickets are some of the highest-volume tickets at any helpdesk — fast, confident handling here matters.

🔐 Account locked out
1
Unlock via Active Directory Users and Computers (ADUC): find the user → right-click → Properties → Account tab → uncheck "Account is locked out," or right-click → Unlock account.
2
Ask if they recently changed their password — old cached credentials on a phone (ActiveSync/Outlook mobile) or a mapped drive retrying the old password repeatedly is the #1 cause of repeat lockouts.
3
Check Event Viewer on the Domain Controller (Security log, Event ID 4740) to find the source machine triggering the lockout — invaluable for repeat lockouts that keep recurring within minutes of unlocking.
4
Common repeat offenders: a forgotten RDP session elsewhere, a scheduled task running under the old credentials, or a saved password in a phone's email/Wi-Fi profile.
🔑 Password reset
1
Verify identity per your org's policy before resetting — this is a classic social engineering target, don't skip verification because someone sounds urgent or important.
2
Reset in ADUC or Microsoft 365 admin center, check "User must change password at next logon" unless org policy says otherwise.
3
If the user is remote and not on VPN, a domain password reset won't sync to a cached Windows login until they connect to VPN or use "Other user" with the new credentials over VPN first — warn them, this trips people up constantly.
4
If using Azure AD/Entra ID with password writeback (hybrid), confirm sync has completed (can take a few minutes) before telling the user to try again.
⏳ Slow logon / "Welcome" screen hangs
1
Roaming/mandatory profiles with large redirected folders (Desktop, Documents) hang while syncing at login — check folder redirection settings and actual folder sizes.
2
Slow or unreachable Domain Controller causes long Group Policy processing — check network path to DC, especially over VPN/site links.
3
Run gpresult /r to see which GPOs are applying and confirm none are obviously misbehaving (e.g. mapping drives to dead file servers, which will time out on every logon).
4
Corrupted local profile — create a new local profile and migrate data, if a specific single user's profile is the issue and others on the same PC log in fine.
🔄 Windows Update stuck or failing
1
Run the built-in Windows Update troubleshooter (Settings → System → Troubleshoot → Other troubleshooters).
2
Restart the Windows Update service: net stop wuauserv then net start wuauserv (admin CMD).
3
If consistently failing with the same error code, rename the SoftwareDistribution folder to force a clean re-download: stop the service, rename C:\Windows\SoftwareDistribution to SoftwareDistribution.old, restart service.
4
Check actual free disk space — updates fail silently/cryptically when there isn't enough room to stage them.
5
Run sfc /scannow and DISM /Online /Cleanup-Image /RestoreHealth if corruption is suspected (random update failures across multiple update cycles).
📁 Mapped network drive missing / disconnected (red X)
1
Confirm the file server is actually reachable: ping servername, then try \\servername\share directly in Explorer.
2
Re-enter credentials if the account password recently changed — mapped drives using stored credentials silently fail after a password reset.
3
If working on-site but not remote, the share likely isn't reachable without VPN — confirm VPN is connected first.
4
Re-map manually to confirm: net use Z: \\server\share /persistent:yes in CMD, swap in actual drive letter/path.

🛡 Security basics for the desk
•
Suspected phishing: don't click anything in the message — check the sender's actual email address (not just display name), hover links to preview the real URL, and report/forward per your org's process.
•
Suspected malware/ransomware: disconnect the machine from the network immediately (pull the cable / disable Wi-Fi) before further investigation — containment first, diagnosis second.
•
Never disclose a user's password to anyone over the phone, including someone claiming to be IT/security/a VIP — verify through a separate channel if anything feels off.
•
USB drives from unknown sources: don't plug into a production machine — use an isolated/sandboxed device if you must inspect one at all.
•
Always lock your own workstation (Win+L) when stepping away from a user's desk while still logged in with elevated credentials.
🖥 Remote support basics
1
RDP (port 3389) needs the target machine's network/firewall to allow it, and the target must have Remote Desktop enabled (Settings → System → Remote Desktop) and the user account added to the Remote Desktop Users group if not an admin.
2
"Remote Desktop can't connect" with a working ping usually means the RDP service isn't listening, the firewall is blocking 3389 specifically, or Network Level Authentication settings mismatch.
3
For end-user screen-share tools (Quick Assist, TeamViewer, etc.), confirm the user has actually granted/accepted control before assuming the connection itself failed.
4
Always tell the user what you're about to do before doing it remotely — especially before closing apps or rebooting, since they may have unsaved work.
📋 Before you escalate a ticket — checklist
A well-documented escalation gets picked up faster and bounces back less
✓
Exact error message (screenshot, not paraphrased) and when it occurs.
✓
What you've already tried, and the result of each step — saves Tier 2 from repeating your work.
✓
Is it one user or many? One device or many? Narrows scope immediately for whoever picks it up next.
✓
When did it start, and did anything change around that time (update, new hardware, moved desks, password reset)?
✓
Device name/asset tag, OS version, and relevant software versions.
✓
Business impact/urgency — "can't process payroll" escalates differently than "icon looks slightly off."
📜 Reading Event Viewer effectively
1
Windows Logs → System: hardware, driver, and OS-level events. Application: software-level crashes/errors.
2
Filter by Event level (Critical/Error/Warning) and a tight time range around when the issue occurred — unfiltered logs are too noisy to read top to bottom.
3
Note the Event ID and Source — searching "[Source] Event ID [number]" usually finds exactly what it means and common fixes.
4
Security log Event ID 4625 = failed logon attempt, 4740 = account locked out, 4624 = successful logon — the three most useful for account-related tickets.
